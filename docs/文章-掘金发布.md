# 我用 NestJS + Vue3 搭了一个 AI 知识库系统，顺便把 RAG 的坑踩了一遍

> 本文记录了一个从零到一的全栈 AI 知识库项目 —— 从「为什么要做」到「怎么做」再到「踩了什么坑」，适合对 RAG 应用感兴趣、正在犹豫要不要自己搞一套的开发者参考。

---

## 🤔 为什么我要自己搭一个知识库？

你有没有这样的经历——

- 丢给 ChatGPT 一篇 50 页的 PDF，它告诉你「我无法阅读这么长的文档」；
- 让 AI 回答一个关于公司内部制度的问题，它一本正经地给你编了个「根据《劳动法》第 88 条」；
- 团队知识散落在飞书文档、Confluence、本地 Word、微信群聊里，问谁谁都不知道在哪。

这不是 AI 不够聪明，而是**它根本没见过你的数据**。

RAG（Retrieval-Augmented Generation）的核心理念很简单：**先找资料，再回答问题**。就像考试前先翻课本再答题，而不是凭记忆瞎蒙。市面上有不少现成的 RAG 产品，但试了一圈下来，我发现它们要么是黑盒 SaaS（数据不在自己手里），要么扩展性差（想换个检索引擎？抱歉不支持），要么就是……太贵了。

所以我决定自己搭一套，名字叫 **Knowledge Quiz 2**（别问为什么叫 2，问就是 1 已经凉了）。

---

## 🎯 核心功能：它能干什么？

### 1. 文档上传与智能解析

支持 **PDF、Word、Excel、PPT、Markdown、TXT、JSON、CSV**，以及 **图片、音频、视频** 和 **网页 URL**。基本上你能想到的格式它都吃——像个毫无挑剔的吃货。

更关键的是，它不是简单地「把文件扔进数据库」，而是做了**结构化解析**：

- PDF 提取每页文本，文字太少时自动触发 OCR；
- Excel 按 Sheet + 行拆分，每一行都是独立可检索的 Chunk；
- PPT 按幻灯片拆分；
- Markdown 按标题层级拆分段落；
- 音频/视频走语音转文字 + 关键帧图像识别双通道。

### 2. AI 智能问答

上传文档后，你可以像跟 ChatGPT 聊天一样提问。系统会：

1. 把你的问题转成向量，在知识库中检索最相关的文档片段；
2. 把检索到的内容 + 对话历史喂给大模型；
3. 流式输出回答，体验丝滑。

### 3. 双路检索 + 重排序

这是这个项目最硬核的部分。检索不是单一通道，而是：

- **向量检索**（Neo4j）：语义相似度匹配，找「意思差不多」的内容；
- **关键词检索**（Elasticsearch）：精确匹配，找「字面上就包含」的内容；
- **RRF 融合**：把两路结果按 Reciprocal Rank Fusion 合并排序；
- **重排序**（Qwen Rerank）：让 AI 再看一遍，挑出真正最相关的。

> 就像招聘流程：简历先过 HR 粗筛（关键词），再过算法匹配（向量相似度），最后技术面试官拍板（重排序）。三层过滤，拒绝平庸。

### 4. 对话记忆

每次对话都有**短期记忆**（当前会话上下文，Redis 存储，1 小时过期）和**长期记忆**（跨会话的用户偏好，Redis 持久化）。AI 不会忘记你上一次聊了什么——至少在它该记得的范围内不会。

### 5. 文档管理

完整的 CRUD：查看文档列表、查看分块详情、编辑单个 Chunk 内容、删除文档或分块。你以为知识库只是上传？不，**数据治理才是长期运营的关键**。

### 前端

- **Vue 3** + Composition API `<script setup>` + TypeScript
- **Pinia** 状态管理
- **UnoCSS** 原子化样式
- **@ai-sdk/vue** 流式对话（这个库是 Vercel AI SDK 的 Vue 版本，专门处理 LLM 流式响应）
- **Lucide Vue** 图标
- **markdown-it + streamdown** 渲染 AI 回复的 Markdown

两个主页面：AI 对话页和后台数据管理页。目前没有用 Vue Router 做路由跳转——而是直接在 App.vue 里切换。够用就好，过度设计是魔鬼。

### 后端

- **NestJS 11**：模块化架构，每个业务领域一个 Module
- **TypeORM** + PostgreSQL：文档、分块、对话、消息的持久化
- **Neo4j**：向量索引存储和相似度检索（1536 维 cosine）
- **Elasticsearch**：全文关键词检索
- **Redis**：缓存和对话记忆
- **RustFS**：S3 兼容的对象存储，存原始文件
- **LangChain + Qwen**：大模型调用（聊天、Embedding、视觉理解）
- **Langfuse**：LLM 调用追踪和可观测性

核心模块一览：

| Module | 职责 |
|--------|------|
| `AiModule` | 聊天、Embedding 生成、视觉理解、流式输出 |
| `DocumentModule` | 文档上传、解析、分块、向量入库 |
| `ConversationModule` | 对话管理、消息持久化 |
| `MemoryModule` | 短期/长期记忆 |
| `RetrievalService` | 双路检索 → RRF 融合 → 重排序 → 上下文扩展 |

### 文件处理流水线

上传一个文件后的完整流程：

```
文件上传 → RustFS 存储 → 文件类型识别 → 文本提取
→ 结构化解析(按页/行/幻灯片/段落) → 分块(RecursiveCharacterTextSplitter)
→ Embedding 生成 → Neo4j 向量入库 + ES 索引 + PostgreSQL Chunk 记录
→ 文档状态更新为 "processed"
```

如果是图片，走视觉模型描述；如果是音频/视频，走语音转文字 + 关键帧识别。**它不是在处理文件，是在消化文件。**

## 🕳️ 踩过的坑与解决方案

### 坑 1：PDF 解析——文字提取是薛定谔的

**问题**：有些 PDF 看着有文字，提取出来全是空白。扫描版 PDF 更是直接交白卷。

**解决**：做了两级 fallback——先用 `pdfjs-dist`（LangChain 的 PDFLoader）提取，如果提取文字少于 20 字符，自动切换到 OCR 模式：先截图每页，再调用视觉模型识别文字。

```typescript
async processPdf(filePath: string): Promise<ParsedDocument> {
  const parsed = await this.loadWithLoader(filePath, (path) => new PDFLoader(path));
  // 提取文字够用就直接返回
  if (parsed.text.replace(/\s+/g, '').length >= 20) return parsed;
  // 不够就 OCR
  return this.ocrPdf(filePath);
}
```

### 坑 2：检索质量——单路检索差点让我放弃

**问题**：纯向量检索会漏掉关键词精确匹配的内容（比如用户问「离职流程」但文档里写的就是「离职流程」这四个字，向量可能因为语义泛化反而排不到前面）。纯关键词检索又会漏掉语义相关但措辞不同的内容。

**解决**：双路检索 + RRF 融合 + 云端重排序。

```typescript
async retrieve(query: string, documentIds?: string[]): Promise<RetrievedChunk[]> {
  const embedding = await this.aiService.generateEmbedding(query);
  // 双路并行检索
  const [vectorHits, keywordHits] = await Promise.all([
    this.neo4jService.search(embedding, 30, documentIds),
    this.elasticsearchService.search(query, 30, documentIds),
  ]);
  // RRF 融合
  const fused = this.fuse(vectorHits, keywordHits);
  // 云端重排序
  const reranked = await this.rerank(query, fused.slice(0, 30));
  // 上下文扩展：补充前后相邻 Chunk
  return this.expandNeighbors(selected, tokenBudget);
}
```

重排序还做了降级处理——如果云端 Rerank API 挂了，自动 fallback 到 RRF 排序，不至于整个问答系统瘫痪。

### 坑 3：旧版 Office 格式——.doc 和 .ppt 是上个世纪的遗产

**问题**：LangChain 的 DocxLoader 只支持 `.docx`，不支持 `.doc`。用户上传一个 `.doc` 文件直接报错。

**解决**：检测到旧格式时，先调 LibreOffice 做格式转换，再走正常解析流程。转换完的临时文件在 finally 里清理掉，不留垃圾。

```typescript
private async convertLegacyOffice(filePath, fileName, targetExtension) {
  const directory = await fs.promises.mkdtemp(path.join(os.tmpdir(), 'rag-office-'));
  await this.execFileAsync('soffice', ['--headless', '--convert-to', targetExtension, '--outdir', directory, sourcePath]);
  // ... 用完即删
}
```

> 这就像遇到老式电视机——你不能直接扔掉，得先找个转换插头才能接到新 HDMI 接口上。

### 坑 4：URL 抓取安全——SSRF 是隐形杀手

**问题**：用户提交一个 URL 让系统抓取内容，但如果这个 URL 指向内网地址（`192.168.x.x`、`127.0.0.1`），就可能触发 SSRF 攻击。

**解决**：在抓取前做 DNS 解析，校验目标 IP 不是私有地址。同时限制最大响应体积和重定向次数。

```typescript
private async assertPublicHost(url: URL): Promise<void> {
  const addresses = await lookup(url.hostname, { all: true });
  if (addresses.some(({ address }) => this.isPrivateAddress(address)))
    throw new Error('禁止访问内网或保留地址');
}
```

### 坑 5：分块策略——Chunk 太大太小都不行

**问题**：Chunk 太大，检索精度下降（一个 Chunk 里塞了三段不同主题的内容）；Chunk 太小，丢失上下文（只返回半句话，AI 根本看不懂）。

**解决**：用了两层策略——

1. **结构化解析优先**：先按文档的自然结构（页、行、幻灯片、段落）拆分，而不是暴力按字数切；
2. **上下文扩展**：检索到相关 Chunk 后，自动补充前后相邻的 Chunk，直到凑够 Token Budget（默认 6000 tokens）。

这样 AI 看到的不是孤零零的一句话，而是有前后文的完整段落。

## ✨ 项目亮点总结

1. **全格式覆盖**：从 PDF 到视频，从 URL 到 JSON，13 种格式一网打尽。文档界的瑞士军刀。
2. **双路检索 + 三层过滤**：向量 + 关键词 + 重排序，RAG 检索质量比单路方案高出一截。
3. **上下文扩展**：不只返回匹配的 Chunk，还自动补充相邻内容，让 AI 回答有据可依。
4. **结构化解析**：尊重文档本身的逻辑结构（标题层级、Sheet、幻灯片），而非暴力切割。
5. **完整的数据治理**：文档和 Chunk 的增删改查全都有，不是「一上传就黑盒」。
6. **全栈 Monorepo**：pnpm workspace 统一前后端，一键 `pnpm install` + `docker compose up` 就能跑起来。
7. **LLM 可观测性**：Langfuse 集成，每次 AI 调用的 Token 消耗、延迟、输入输出都有记录，调优不靠猜。
8. **优雅降级**：OCR fallback、Rerank fallback、旧格式转换——每一条可能失败的路都有 Plan B。
