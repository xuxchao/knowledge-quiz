# 云数据库迁移 FAQ — 问答问题集 (Q&A)

> **配套文档**：`cloud-db-faq.md`  
> **难度分层**：⭐ 事实型 | ⭐⭐ 理解型 | ⭐⭐⭐ 应用型

## 事实型问题 (Factual Questions)

**Q1.** 自建 MySQL 5.7 迁移到云数据库时，常见的字符集风险是什么？  
<details><summary>答案</summary>自建库常用 `utf8`（3字节），云数据库默认 `utf8mb4`（4字节），可能导致索引超长。</details>

**Q2.** 逻辑备份（mysqldump）适合多大数据量的迁移？  
<details><summary>答案</summary>小数据量（< 10GB）。</details>

**Q3.** DTS 迁移中遇到主键冲突应如何解决？  
<details><summary>答案</summary>设置冲突策略为"覆盖"或"忽略"。</details>

**Q4.** 云数据库高可用方案中，"跨可用区主备"和"单可用区主备"的 RTO 分别是多少？  
<details><summary>答案</summary>两者都是 < 60s。</details>

**Q5.** 备份策略建议中，全量备份和增量备份分别保留多少天？  
<details><summary>答案</summary>全量备份保留最近 7 天，增量备份保留最近 3 天。</details>

**Q6.** Buffer Pool 命中率低于多少时需要关注内存不足问题？  
<details><summary>答案</summary>低于 95%。</details>

## 理解型问题 (Comprehension Questions)

**Q7.** 为什么 DTS 在线迁移的"数据一致性"是"最终一致"而非"强一致"？  
<details><summary>答案</summary>DTS 通过解析源库 binlog 实现增量同步，在增量同步过程中源库和目标库之间存在秒级的时间窗口差异。DTS 保证最终目标库数据与源库一致，但不保证在任意时刻两者完全相同。通过全量对账可以在任务完成后验证最终一致性。</details>

**Q8.** 解释"两地三中心"方案的 RPO 为什么能达到 0，而"跨地域灾备"只有 < 1s？  
<details><summary>答案</summary>两地三中心通常在同城双活+异地灾备的基础上实现同步复制，写操作在所有副本确认后才返回成功，因此 RPO=0（零数据丢失）。跨地域灾备受光速和网络延迟限制，通常使用异步复制，存在最坏情况下丢失数秒数据的可能。</details>

**Q9.** 迁移前为什么要执行 `information_schema.TABLES` 查询非 InnoDB 引擎的表？  
<details><summary>答案</summary>云数据库可能禁用或限制 MyISAM 等存储引擎。如果存在非 InnoDB 引擎表，需要提前转换为 InnoDB。MyISAM 不支持事务、行锁、崩溃恢复等特性，生产环境中已不推荐使用。</details>

**Q10.** 为什么索引数量建议不超过 5 个？  
<details><summary>答案</summary>每个索引在数据写入（INSERT/UPDATE/DELETE）时都需要同步维护，索引越多写入性能越低。同时也会占用更多磁盘空间和 Buffer Pool 内存。5 个是经验值，实际根据业务查询模式权衡。</details>

## 应用型问题 (Application Questions)

**Q11.** 一个日均 500 万订单的电商平台的 MySQL 实例需要迁移上云，要求停机时间 < 5 分钟。请设计完整的迁移方案。  
<details><summary>答案</summary>使用 DTS 在线迁移：1) 预检查：兼容性、数据规模、带宽评估；2) 创建目标云数据库实例并配置参数；3) 启动 DTS 全量+增量迁移任务；4) 等待增量延迟降至 < 1 秒；5) 在低峰时段（凌晨 3-4 点）执行切换：停止源库写入 → 等待增量追平 → 修改 DNS/配置指向云数据库 → 恢复写入；6) 监控 30 分钟确认稳定后关闭迁移任务。</details>

**Q12.** 迁移后发现 P99 查询延迟从 20ms 上升到 80ms。请列出排查思路和优化方向。  
<details><summary>答案</summary>排查：1) 检查网络延迟（应用与数据库是否在同一 VPC/可用区）；2) 分析慢查询日志，对比迁移前后执行计划差异；3) 检查实例规格是否匹配（CPU、内存、磁盘 IOPS）；4) 检查参数组配置是否与自建库一致（如 sort_buffer_size、join_buffer_size）；5) 检查数据库连接池配置是否匹配。优化：调整实例规格、创建缺失索引、配置只读实例读写分离。</details>

**Q13.** 公司 CFO 要求将云数据库成本降低 30%。请根据文档内容制定优化方案。  
<details><summary>答案</summary>组合方案：1) 计费模式切换：短期稳定业务改为包年包月（节省 30-50%）；2) 部署只读实例实现读写分离，降低主实例规格（节省 20-35%）；3) 数据生命周期管理：6 个月以上历史数据归档至对象存储（节省 15-25%）；4) 清理无用数据和重复索引；5) 非生产环境（dev/staging）使用弹性伸缩，按需启停。</details>

**Q14.** 为什么迁移过程中需要在地域选择上关注"同地域同可用区"？如果生产环境必须跨地域部署数据库，应如何保障性能？  
<details><summary>答案</summary>同地域同可用区可将网络延迟控制在亚毫秒级，最高效。跨地域会增加 10-50ms 延迟。保障方案：1) 应用层使用连接池减少连接建立开销；2) 使用只读实例在应用所在地域就近读取；3) 写操作通过专线/VPN 降低不可靠网络的影响；4) 应用层引入缓存（Redis）减少数据库查询次数。</details>

**Q15.** 一个 SaaS 平台需要为每个租户提供独立的数据库实例。请设计一套高效的批量迁移策略和后续运维方案。  
<details><summary>答案</summary>迁移：1) 编写自动化脚本根据租户列表批量创建目标实例；2) 按租户优先级分批迁移（P0 先迁、P1/P2 后续）；3) 每个租户独立 DTS 任务并行执行，设置并发上限控制负载。运维：1) 监控所有租户实例的统一仪表盘；2) 备份策略模板化，自动应用到新实例；3) 实例参数通过配置管理平台统一管理；4) 升版升级采用金丝雀策略（先 10% 租户验证）。</details>
